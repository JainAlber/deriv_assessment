# Tech Stack

Everything used by the Financial Content Intelligence Pipeline, what it does, and why it was picked.

---

## Language and runtime

| Item | Version | Purpose |
|---|---|---|
| **Python** | 3.10+ | Main language. Uses modern typing syntax (`list[str]`, `str \| None`). |
| **Windows / PowerShell** | 5.1+ | Development environment for this build. The code itself is OS-agnostic. |

Why Python: rich ecosystem for HTTP, HTML parsing, and LLM SDKs; trivial to read/maintain.

---

## LLM provider

| Item | Purpose |
|---|---|
| **Groq Cloud** | LLM inference provider. Chosen for speed (sub-second latency on 70B models) and competitive pricing. |
| **`llama-3.3-70b-versatile`** | The model that does entity extraction, sentiment scoring, QA review, and report writing. Picked for its strong instruction-following and reliable JSON-mode output. |
| **`groq` Python SDK** | `pip install groq`. The official client. We call `client.chat.completions.create(...)` with `response_format={"type":"json_object"}` for structured stages. |

Pricing baseline used in `cost_report.json`:
- Input: $0.59 / 1M tokens
- Output: $0.79 / 1M tokens

These are stored as constants and easy to update if Groq changes prices.

---

## Web scraping

| Item | Purpose |
|---|---|
| **`requests`** | Standard HTTP client. Used with a 20-second timeout and a desktop User-Agent to fetch each source URL. |
| **`beautifulsoup4`** | HTML parser. We use `BeautifulSoup(html, "html.parser")` — no extra C-extensions, no `lxml` dependency. |
| **Built-in `re`** | Regex used for extracting numerical signals (`PRICE_PATTERN`) and recovering JSON from imperfect LLM output. |

A real-browser User-Agent is set on every request because Yahoo Finance, Reuters, and similar sites return 403 / interstitials to default Python UA strings.

---

## Configuration and secrets

| Item | Purpose |
|---|---|
| **`python-dotenv`** | Loads `.env` so `GROQ_API_KEY` becomes available via `os.getenv`. |
| **`.env`** | Holds the API key. Listed in `.gitignore` so it never reaches version control. |
| **`.env.example`** | Template committed to the repo. Users copy → `.env` and fill in. |
| **`sources.json`** | The list of finance URLs the pipeline ingests. The only "data input" of the system. |

## Standard-library modules used

| Module | Why |
|---|---|
| `json` | Read `sources.json`, write every output artifact. |
| `os` | `os.getenv` for the API key. |
| `re` | Regex parsing of prices and recovery of malformed LLM JSON. |
| `sys` | Process exit codes for `pipeline.py` and `validate.py`. |
| `time` | Per-source latency timing. |
| `hashlib` | SHA-1 for content_ids, SHA-256 for prompt hashes. |
| `traceback` | Final-resort error printing in `main()`. |
| `pathlib` | All file paths use `Path` rather than string concat. |
| `urllib.parse` | `urlparse(url).netloc` to derive `source_name`. |
| `datetime` (with `timezone.utc`) | ISO-8601 timestamps everywhere. |

No third-party dependency is added when a stdlib option is sufficient.

## Data formats

| Format | Where it's used |
|---|---|
| **JSON** | All structured outputs (`*.json` files). Pretty-printed with `indent=2`, UTF-8, `ensure_ascii=False` so non-ASCII names round-trip. |
| **JSONL** | `llm_calls.jsonl` — one JSON object per line, append-only audit log. JSONL chosen over JSON-array because it stays valid even if the run crashes mid-write. |
| **Markdown** | `reports/*.md` — Markdown is human-readable on disk, renders nicely in any viewer, and is what the reports' downstream consumers (PMs, traders) expect. |

```
requests>=2.31.0

```

Install with:
```
pip install -r requirements.txt
```

Versions are pinned with `>=` rather than `==` so security updates are picked up automatically; the floors are set to the versions known to support the APIs used here.

## Architectural choices

| Choice | Why |
|---|---|
| **Single-file pipeline** | Easy to audit and easy to run. The whole flow fits in one file you can read top-to-bottom. |
| **One LLM call per entity for sentiment** | Required for entity-specific sentiment. Sending all entities at once would let the model collapse into page-level scoring. |
| **JSON-mode on Groq** | Forces the model to return structured output, removing fragile regex parsing for the structured stages. |
| **Stable IDs (`content_id`, `entity_id`, `issue_id`)** | Lets every artifact cross-reference every other artifact without ambiguity. |
| **Hybrid QA (rules + LLM)** | Deterministic Python catches the easy stuff (low confidence, duplicates, sign flips); the LLM catches subtle issues like ungrounded claims. |
| **Audit log per LLM call** | `llm_calls.jsonl` is append-only and contains hash, tokens, timestamps, and errors — enough to reconstruct any run for debugging or billing reconciliation. |
| **Disclaimers on every report** | The product is "market intelligence"; calling it "financial advice" is a regulatory minefield. The disclaimer is hard-coded. |

## Things deliberately NOT used

| Skipped | Reason |
|---|---|
| **`lxml`** | `html.parser` ships with Python and is enough for our generic scrape; one fewer compiled dependency. |
| **`scrapy` / Selenium / Playwright** | Adds infrastructure for sites that we successfully scrape with plain `requests`. If the source list grows to JS-heavy SPAs, a Playwright fallback could be added behind an interface. |
| **Async (`asyncio` / `aiohttp`)** | Source count is small (5). Sequential `requests` keeps the code simple and easier to debug. Easy to swap in later if throughput becomes an issue. |
| **Database** | Outputs are flat JSON files. A DB adds operational burden for a single-shot pipeline. Easy to import the JSON later if needed. |
| **Custom NER models** | The LLM does NER, alias resolution, and sentiment in two prompts. Training a domain NER model would be a multi-week project; the LLM gets us there in one afternoon. |
| **A web framework** | Not a service — a pipeline. Wrapping it in FastAPI / Flask is a future step, not a current need. |
